## 二、产品定位（本次新增）

### 核心用户需求
"今天不知道穿什么" — 这是核心场景，不是单品浏览，不是电商搜索。

### 产品形态决策
- **不做**纯页面推荐瀑布流（拼不过ASOS/Nordstrom的数据量）
- **核心**做对话式AI推荐：一个大输入框，用户描述需求，下方出完整搭配方案
- **不做**聊天气泡界面（像客服），做输入框+结果页（像工具）
- 底部保留"Today's Picks"瀑布流给不想打字的用户兜底
- 页面底部放快捷标签：Casual / Work / Date Night

### APP导航结构（来自功能文档）
Home | Inspiration | Add | AI Stylist | Profile

## 三、推荐模型在APP里的位置（本次新增）

### 老板确认的工作方式
推荐模型配合Helen的穿搭模型使用，流程是：
**单品推荐 → 搭配全身衣服 → VTO上身确认**
不单独做单品推荐展示。

### 对应架构图三层
- Output Layer：Daily Recommendation / VTO / Stylist Chatbot
- Model Layer：LLM Clothes Matching + Recommendation Engine + VTO Engine
- Data Layer：User Profile / Behavior Data / Fashion Guides / Visual Data

### 推荐模型具体使用位置
1. **首页**：按5个场景分类推荐完整搭配
   - Everyday/Casual / Events/Going Out / Work/Office / Active/Outdoor / Travel
2. **Swap替换**：用户不喜欢某件单品，局部重新召回同类替代品
3. **Inspiration页**："For You" tab个性化推荐
4. **AI Stylist Chat**（2.0）：意图解析Prompt在这里发挥作用

## 四、Work/Office场景完整传导路径（本次新增）

```
用户触发（点击Work/Office tab）
    ↓
上下文信号读取（天气今日/明日高低温 + 日历事件）
    ↓
Step1 意图解析
  occasion → ["Work / Office"] 或 ["Business Formal"]
  结合天气调整season，结合日历调整正式程度
    ↓
Step2 验证层（枚举值校验 + JSON修复）
    ↓
Step3 衣橱过滤
  "我的衣橱" vs "购买推荐" 两条路分开
    ↓
Step4 向量检索（Vertex AI，返回前10候选）
  过滤exclude条件 + 用户dislike黑名单
    ↓
Step5 冷启动判断
  新用户 → 热门职场款兜底
  老用户 → 正常检索流程
    ↓
Step6 精排LLM（Flash-Lite，选2-3套）
  输出 reason + styling_tip + outfit_summary
    ↓
结果展示
  "我的衣橱搭配"（已有单品组合）
  "购买推荐"（附购买链接）
  "推荐理由卡片"
    ↓
用户交互
  Swap替换 → 局部重新召回（同category，不重跑意图解析）
  分享穿搭 → 附购买链接，图片下生成文字可编辑
  VTO试穿 → 上身效果确认（末端可选步骤）
  反馈收集 → 点踩/保存/跳过 → Firestore
    ↓
Time Capsule（保存今日造型，记录穿搭备注）
    ↓
反馈闭环（L1点踩即时屏蔽，L2每日权重更新）
```

### Swap替换技术细节
- 不重跑意图解析，复用当前搭配的occasion/style_tags参数
- 限定category必须和被替换单品一致（衬衫换衬衫）
- 两个入口：
  - "从衣橱换"：优先找用户已有单品
  - "找平替"：price_tier ≤ 原单品，按价格升序排列
- 精排只选1件（不是2-3套），速度更快

## 五、VTO模块（本次新增）

### 市场调研结论
| APP | 核心 | VTO类型 |
|---|---|---|
| Alta | 数字衣橱+搭配 | SAM抠图+自研模型，非API |
| Whering | 数字衣橱+搭配 | Avatar形式，非真实试穿 |
| Style DNA | 色彩/体型分析+推荐 | 无VTO |
| Google Shopping | 电商搜索 | 自研Nano Banana，效果最好 |
| FASHN | 专门做VTO | 真实试穿，对外API |

**结论**：Alta等头部APP效果好是因为自建模型+专属数据，不是API。
我们前中期老老实实用FASHN API是正确路线。

### VTO输入规范（两张图原则）
1. **Person Image**：正面站立，手臂自然下垂，背景干净纯色
2. **Garment Image**：平铺/挂拍白底图，无人穿着，轮廓展平
- 禁止：把"模特穿A衣服的图"当garment image（会造成纹理融合灾难）

### 质量控制：静默降级机制
不做"分批上线"，而是：
- FASHN返回图片后，后台跑质量检测
- 达标 → 展示试穿效果图
- 不达标 → 静默回退到商品原图，用户感知不到失败
- 失败率统计：硬失败（API报错）和软失败（效果差）分开记录

### 失败率判断标准
- 硬失败率 > 5% → 商品图质量有问题，检查输入
- 软失败率 > 20% → 值得加SAM3前处理
- 软失败率 < 10% → FASHN默认处理够用，不用动
- 数据来源：上线后前两周500-1000条真实请求

### VTO前中后期路线

**前期（现在）— FASHN API**
- 注册fashn.ai账号，获取API Key
- 手动测试5-10件商品，记录各品类效果好坏
- 工程师接入约1-3天，$0.075/张
- 静默降级机制上线，👍👎反馈按钮收集软失败信号

**中期（软失败率>20%时触发）— 加SAM3前处理**
- Meta SAM3做商品图抠图（从同事处获取脚本或自建）
- SCHP做人体解析，生成Mask告知FASHN替换哪里
- GCP CPU机器即可，工程师约3-5天开发量
- API调用降低20-30%

**后期（日均10万张触发）— 自建模型**
- FASHN开源版部署到GCP A100/L4
- 月固定$1,500-3,000，不随请求量涨价
- 针对用户群专项调优

### VTO费用估算
50万用户×10%使用VTO×每天1件 = 每天5万张
月费约$6,000-9,000（$0.04-0.075/张）

## 十二、APP功能规划摘要（来自功能文档）

### 1.0优先功能
- 首页：天气集成（今日/明日高低温）+ 5场景个性化推荐 + 衣橱/购买分开展示

- 3D渲染/正侧背三视图
- AI Stylist Chat（了解用户体型/品牌/尺码）
- 小票导入商品
- Closet Compatibility Check（线下扫码判断是否重复）

### 3.0规划功能
- 手机传送穿搭（含购买链接）

### 关键产品决策
- 衣橱和Wishlist明确分开（已有 vs 想要）
- 展示整套穿搭，不展示单品（Inspiration页除外，用于转化）
- "For You"在上，"Trending"在下（Alta已在测试同方案）
